# 提案：_ 弃元

## 动机

有两个独立但相关的需求：

**1. 忽略返回值**

调用有副作用且有返回值的函数时，有时不需要返回值。目前没有语法表达"明确丢弃"：

```coflow
var _ = queue.pop()     # 弹出但不使用
var _ = table.remove(key)
```

虽然直接写 `queue.pop()` 作为语句也可以，但 `var _` 明确表达了"我知道有返回值，我选择忽略"的意图，有助于静态分析工具区分"忘记用"和"故意忽略"。

**2. 解构时忽略某些位置**

与解构提案联动，`_` 用于跳过不需要的部分（目前解构尚未进入核心，此处作为占位）：

```coflow
for _, v in scores { }         # 只关心 value
var [_, second, _] = items     # 只取第二个
```

## 语法

`_` 是特殊标识符，在以下位置有特殊语义：

1. 变量声明左侧：`var _ = expr` — 求值 expr，丢弃结果
2. 赋值左侧：`_ = expr` — 同上
3. 函数参数名：`fn f(_: int)` — 接受参数但不使用
4. 解构位置：见解构提案

```coflow
var _ = side_effect()
_ = update_state()

fn on_event(event: any, _: int) {
  process(event)
}
```

## 语义

- `_` 不绑定任何值，不能被读取：`var _ = 1; print(_)` 是错误。
- 多个 `_` 在同一作用域内不冲突：每次出现都是独立的弃元位置。
- `var _` 仍然要求右侧表达式被完整求值（包括副作用）。

## 与现有语法的关系

`_` 目前是合法的普通标识符（满足 XID_Start 规则中的 `_` 例外）。引入弃元语义是一个**软语义约束**，不需要修改 lexer（`_` 仍然是 `Ident` token）：

- Parser 层：无需特别处理，`_` 走普通 `Ident` 路径。
- 语义层：识别名字为 `"_"` 的 `VarDecl`，标记为弃元，禁止读取，不报"未使用变量"警告。

## 实现成本

极低。

- Lexer / AST / Parser：无变化。
- 语义层：在 `VarDecl` 和赋值目标的名字检查中，对 `"_"` 特殊处理。

## 开放问题

1. `_` 可以出现多次（`var _ = 1; var _ = 2`），这是否允许？建议允许，弃元不参与重复定义检查。
2. `__` / `_x` 等以下划线开头的标识符是否有特殊含义？建议只有单独的 `_` 是弃元，其他仍是普通标识符。
3. 是否需要在 `_` 出现在可读位置时给出编译警告而非硬错误？
