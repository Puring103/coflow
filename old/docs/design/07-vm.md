# coflow VM 执行模型设计

本文描述 coflow 字节码虚拟机的执行模型，是 VM 实现的架构依据。指令集定义见 `docs/design/bytecode.md`。

定位：

1. 寄存器式解释器，第一版纯解释执行
2. 第二版视 profile 数据考虑 JIT 编译热函数
3. 目标：游戏 / 配置脚本的宿主嵌入场景，低延迟、可控内存

## Value 表示

Value 是 VM 中流转的基本单元，设计方案影响整体内存带宽与分支开销。有两种主要方案。

### 方案 A：Tagged union（Rust enum）

```rust
enum Value {
    Null,
    Bool(bool),
    Int(i64),
    Float(f64),
    String(Rc<str>),
    Array(Rc<RefCell<Vec<Value>>>),
    Dict(Rc<RefCell<Vec<(Value, Value)>>>),
    Object(Rc<RefCell<Object>>),
    Closure(Rc<Closure>),
    Iterator(Rc<RefCell<dyn Iterator>>),
    // ...
}
```

优点：实现直观，Rust 类型系统全覆盖，无 unsafe。

缺点：每个 `Value` 在 64 位平台上占 16–24 字节（tag + 最大 payload 对齐），寄存器数组内存占用较大，cache miss 压力较高。

### 方案 B：NaN tagging

利用 IEEE 754 双精度浮点 NaN 空间（quiet NaN 的低 51 位）编码非浮点值，使每个 `Value` 恰好占 8 字节：

```
浮点数：正常 f64（非 NaN）
NaN 模式：高 13 位全为 1，低 51 位编码 tag + payload
  tag 3 位：Null / Bool / Int / Pointer
  payload 48 位：int 值或堆指针（48 位地址空间足够用）
```

优点：`Value` 紧凑，寄存器数组 cache 友好，解释器循环内 Value 传递只需 64 位整数拷贝。

缺点：需要 unsafe，负整数编码需要特殊处理，调试复杂度较高。

### 决策

**第一版采用方案 A（tagged union）**，优先保证正确性与可维护性。待管线跑通、有 benchmark 数据后，第二版在性能敏感场景切换到 NaN tagging，两者对外接口相同，可以局部替换。

## 栈帧布局

每次函数调用对应一个 `CallFrame`：

```rust
struct CallFrame {
    proto: Rc<Proto>,           // 当前函数的字节码与常量池
    pc: usize,                  // 程序计数器（指向 proto.code 的索引）
    regs: Box<[Value; 256]>,    // 寄存器窗口，256 个寄存器
    upvalues: Vec<Rc<UpvalueBox>>, // 当前闭包捕获的 upvalue
    open_upvalues: Vec<(usize, Rc<UpvalueBox>)>, // 尚未关闭的 upvalue，索引为寄存器号
    handler_stack: Vec<ExceptionHandler>, // 异常 handler 栈
    is_generator: bool,         // 是否是 generator 帧
}

struct ExceptionHandler {
    handler_pc: usize,
    frame_depth: usize,  // 栈展开用
    err_reg: u8,
}
```

调用帧通过 `Vec<CallFrame>` 组成调用栈。帧之间不共享寄存器数组，被调函数的 reg[0] 是其第一个参数（调用方在 CALL 前已将参数写入连续寄存器，VM 将其拷贝到新帧的 reg[0..argc]）。

寄存器窗口固定 256 个，`Box<[Value; 256]>` 堆分配，避免帧大小影响 Rust 调用栈。

## 执行循环

### 基础版（match dispatch）

```rust
loop {
    let instr = frame.proto.code[frame.pc];
    frame.pc += 1;
    match decode_opcode(instr) {
        OpCode::Move => { ... }
        OpCode::LoadK => { ... }
        OpCode::Add => { ... }
        // ...
    }
}
```

每条指令的处理逻辑内联在 `match` 分支中，Rust 编译器可将其优化为跳转表。

### 第二版（computed goto / unsafe 跳转表）

Rust 没有原生 computed goto，但可以通过 unsafe 跳转表模拟：

```rust
static DISPATCH_TABLE: [unsafe fn(&mut Vm, u32); 256] = [ ... ];

loop {
    let instr = frame.proto.code[frame.pc];
    frame.pc += 1;
    let handler = DISPATCH_TABLE[opcode_of(instr) as usize];
    unsafe { handler(self, instr) };
}
```

第一版不做此优化，先保证正确性，性能数据驱动后续决策。

## Generator（iter fn）执行

`iter fn` 调用时不立即执行函数体，而是创建一个挂起的 generator 对象：

```rust
struct Generator {
    frame: Option<SuspendedFrame>,  // None 表示已完成
    state: GeneratorState,
}

enum GeneratorState {
    NotStarted,
    Suspended,
    Running,
    Completed,
}

struct SuspendedFrame {
    proto: Rc<Proto>,
    pc: usize,
    regs: Box<[Value; 256]>,
    upvalues: Vec<Rc<UpvalueBox>>,
    open_upvalues: Vec<(usize, Rc<UpvalueBox>)>,
    handler_stack: Vec<ExceptionHandler>,
}
```

`next()` 调用时将 `SuspendedFrame` 恢复为活跃的 `CallFrame` 推入调用栈，继续执行直到遇到 YIELD 或 RETURN：

- **YIELD rA**：将 `reg[rA]` 作为本次 step 的 value，把当前帧序列化回 `SuspendedFrame`，返回 `{ value, done: false }`
- **RETURN**：generator 完成，销毁帧，返回 `{ value: null, done: true }`

与普通函数帧的区别：

- generator 帧在 YIELD 时不弹出，而是序列化到堆上保留
- 普通函数帧在 RETURN 时立即弹出并销毁
- generator 帧的 `open_upvalues` 在挂起期间保持存活，upvalue box 仍有效

## Upvalue 管理

Upvalue 实现闭包对外层变量的捕获，分两个阶段：

### Open upvalue

变量仍在栈上（外层函数仍在执行）时，upvalue 持有对栈上寄存器的引用：

```rust
struct UpvalueBox {
    state: UpvalueState,
}

enum UpvalueState {
    Open { frame_idx: usize, reg: u8 },  // 指向调用栈某帧的某个寄存器
    Closed(Value),                        // 值已移入堆
}
```

同一个寄存器被多个闭包捕获时，它们共享同一个 `Rc<UpvalueBox>`。

### 关闭 upvalue（CLOSE 指令）

作用域退出时 codegen 发出 `CLOSE rA`，VM 将从 `rA` 开始的所有 open upvalue 转为 closed 状态：将寄存器中的当前值拷入 `UpvalueBox::Closed(value)`，寄存器本身不再被引用。

多个闭包共享同一个 `Rc<UpvalueBox>`，关闭后所有持有者都能通过 box 访问到同一个堆上的值：

```rust
// 共享同一个 Rc<UpvalueBox>
let box1: Rc<UpvalueBox> = ...;
let box2 = Rc::clone(&box1);
// close 后 box1 和 box2 都看到 Closed(value)
```

Generator 挂起时，其 open upvalue 保持 Open 状态，不提前关闭；外层函数退出时才触发关闭。

## 异常处理

### Handler 栈

每个调用帧维护一个 `handler_stack`。`TRY rA, sBx` 压入新 handler，`ENDTRY` 弹出：

```
handler = ExceptionHandler {
    handler_pc: current_pc + sBx,
    frame_depth: call_stack.len(),
    err_reg: rA,
}
```

### 抛出与栈展开

`THROW rA` 执行时：

1. 取错误值 `err = reg[rA]`
2. 向上搜索 handler：先在当前帧的 `handler_stack` 查找；无则弹出当前帧，继续向上
3. 找到 handler 后，将调用栈回退到 `handler.frame_depth`（弹出中间所有帧）
4. 将 `err` 写入 handler 帧的 `reg[handler.err_reg]`
5. 跳转到 `handler.handler_pc` 继续执行

若搜索到调用栈底部仍未找到 handler，报"未捕获异常"并终止执行。

### 错误对象

错误对象是一个普通 Object，标准字段为：

- `message: string`：错误描述
- `stack: string`：调用栈快照（由 THROW 时填充，格式由宿主决定）

宿主可通过 Host API 构造自定义错误对象，只要包含 `message` 字段即可被 catch 捕获。

## 模块加载

模块加载分三阶段，与 sema 阶段对应：

### 阶段一：声明收集

解析模块顶层的所有 class / enum / config / var / fn 声明，注册到模块符号表（`ModuleSymbols`）。此阶段不执行任何代码，只收集名字和类型信息。对应 sema 的 P1。

### 阶段二：配置常量求值

对所有 `config` 声明按拓扑序求值，产出深只读的配置值表（`config_values`）。此阶段在 HIR 求值器上运行，不进入 VM 字节码执行路径。对应 sema 的 P3。

### 阶段三：var 初始化

对所有 `var` 声明按声明顺序执行初始化表达式。此阶段进入 VM 字节码执行，可以调用 Host API（如读取外部数据、调用宿主函数）。VM 在一个专用的"模块初始化帧"中执行，完成后销毁帧，全局变量表保留结果。

跨模块加载（`import`）在第一版不实现，`import` 声明注册为占位符，访问时报"未实现"。

## GC 策略

### 第一版：引用计数（`Rc`）

所有堆分配对象（String、Array、Dict、Object、Closure、UpvalueBox）使用 `Rc<T>`（单线程）或 `Arc<T>`（多线程宿主）包装：

- 优点：析构确定性高，无 stop-the-world 停顿，适合游戏场景
- 缺点：循环引用导致内存泄漏

循环引用处理策略：

1. **weak reference**：对可能形成循环的字段（如父子对象相互引用）使用 `Weak<T>`，由业务层决定哪一侧持弱引用
2. **手动打破**：提供 `GcHandle::break_cycle()` 宿主 API，允许宿主在对象销毁前显式切断引用
3. 第一版不实现自动循环检测，文档说明循环引用限制

### 第二版：Tracing GC（待定）

视内存泄漏问题严重程度决定是否引入 tracing GC（如标记-清除或增量 GC）。第二版如引入，`Value` 内部的 `Rc` 替换为 GC 管理的指针，对外 API 不变。

## Host API 接口

宿主通过注册 Rust 函数向脚本暴露能力。函数签名：

```rust
type HostFn = fn(&mut VmContext, &[Value]) -> Result<Value, RuntimeError>;

struct RuntimeError {
    message: String,
    // 可扩展：error_code, cause 等
}
```

注册方式：

```rust
vm.register_fn("print", |_ctx, args| {
    for arg in args {
        print!("{}", arg.to_display());
    }
    println!();
    Ok(Value::Null)
});
```

`VmContext` 暴露有限的 VM 状态供宿主函数使用：

```rust
struct VmContext<'vm> {
    globals: &'vm mut GlobalTable,
    // 未来可扩展：堆分配、模块加载回调等
}
```

宿主函数抛出 `RuntimeError` 时，VM 将其包装成脚本侧的 error 对象（填充 `message` 和 `stack`），走正常 THROW 流程，可以被脚本的 try/catch 捕获。

内建函数（`error` / `iter` / `range` / `print`）以相同接口注册为特殊的内建 `HostFn`，在 VM 初始化时注入全局符号表。
