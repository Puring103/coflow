# CFD V2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 CFD 文本格式迁移到新的 typed reference 语义，并补齐同类型记录分组、`...` 覆盖语法和正式 spec。

**Architecture:** `coflow-data-model` 继续作为所有来源的语义归一层，新增 object/dict spread 输入值并在数据模型验证阶段展开。`coflow-cfd` 只负责 schema-guided 文本解析，把 `&key`、`@Type.key`、`@Type.key.path[index]`、分组记录和 `...` 语法转换为 source-neutral input values。Excel / cell-value 现有引用语义保持不变。

**Tech Stack:** Rust workspace, `coflow-cft`, `coflow-data-model`, `coflow-cell-value`, `coflow-cfd`, Cargo test.

---

## 文件职责

- `crates/coflow-data-model/src/model.rs`：新增 `CfdInputValue` spread 表达和构造函数。
- `crates/coflow-data-model/src/compiler.rs`：在验证阶段将 object/dict spread 合并成 `RecordDraft` / `CfdValueDraft::Dict`，保留类型检查、引用解析和默认值规则。
- `crates/coflow-data-model/src/schema_view.rs`：更新 `input_value_kind`，保证诊断文本能识别 spread。
- `crates/coflow-data-model/tests/model.rs`：新增 data-model spread 行为测试。
- `crates/coflow-cfd/src/lib.rs`：迁移 CFD 文本语法：无 `=`、逗号分隔、无记录分号、`#` 注释、typed references、分组记录和 `...`。
- `crates/coflow-cfd/tests/cfd_loader.rs`：用新语法重写 CFD 测试并覆盖兼容性边界。
- `docs/spec/12-cfd.md`：新增 CFD 设计 spec，记录语法、引用、覆盖、分组、与 Excel/cell-value 的边界。

```rust
#[test]
fn object_spread_merges_record_refs_before_local_overrides() {
    let schema = compile_schema(
        r#"
            type Stats { hp: int; attack: int; }
            type Monster { name: string; stats: Stats; }
        "#,
    );

    let mut builder = CfdDataModel::builder(&schema);
    builder.add_record(
        "base",
        "Monster",
        [
            ("name", CfdInputValue::from("Base")),
            (
                "stats",
                CfdInputValue::object_with_declared_type([
                    ("hp", CfdInputValue::from(100_i64)),
                    ("attack", CfdInputValue::from(20_i64)),
                ]),
            ),
        ],
    );
    builder.add_record(
        "elite",
        "Monster",
        [
            (
                "stats",
                CfdInputValue::object_spread(
                    [CfdInputValue::path_ref(
                        "Monster",
                        "base",
                        [CfdRefPathSegment::Field("stats".to_string())],
                    )],
                    [("hp", CfdInputValue::from(180_i64))],
                ),
            ),
            ("name", CfdInputValue::from("Elite")),
        ],
    );

    let model = builder.build().expect("spread should build");
    let elite_id = record_id_at(&model, 1);
    let Some(CfdValue::Object(stats)) = model
        .record(elite_id)
        .and_then(|record| record.field("stats"))
    else {
        panic!("expected stats object");
    };
    assert_eq!(stats.field("hp"), Some(&CfdValue::Int(180)));
    assert_eq!(stats.field("attack"), Some(&CfdValue::Int(20)));
}
```

```rust
                )],
                [(
                    CfdInputDictKey::enum_variant("Element", "Fire"),
                    CfdInputValue::from(20_i64),
                )],
            ),
        )],
```

```rust
    entries: Vec<(CfdInputDictKey, CfdInputValue)>,
```

- [ ] **Step 2: 更新诊断 kind**

`input_value_kind` 对 `ObjectSpread` 返回 `"object spread"`，对 `DictSpread` 返回 `"dict spread"`。

- [ ] **Step 3: 在验证阶段展开 object spread**

在 `validate_value` 的 `CfdType::Type(expected)` 分支中同时接受 `Object` 和 `ObjectSpread`。实现目标：

- spread value 必须在同一个 expected object 类型上下文下验证。
- spread 可以是 `RecordRef`、`PathRef`、inline object 或另一个 `ObjectSpread`。
- spread 结果必须是 object draft。
- 多个 spread 按出现顺序合并，后者覆盖前者。
- 本地字段覆盖所有 spread 字段。
- 本地字段仍走现有字段校验、默认值和 required 规则。

- [ ] **Step 4: 在验证阶段展开 dict spread**

在 `CfdType::Dict(key_ty, value_ty)` 分支中同时接受 `Dict` 和 `DictSpread`。实现目标：

- spread value 必须在同一个 dict 类型上下文下验证。
- spread 可以是 `PathRef`、inline dict 或另一个 `DictSpread`。
- 多个 spread 按出现顺序合并，后者覆盖前者。
- 本地 entries 覆盖 spread entries。
- 本地 entries 之间仍然检测重复 key。

### Task 4: 实现 CFD v2 parser

**Files:**
- Modify: `crates/coflow-cfd/src/lib.rs`
- Test: `crates/coflow-cfd/tests/cfd_loader.rs`

- [ ] **Step 1: 顶层记录解析**

`parse_records` 支持两种形式：

- `key: Type { fields }`
- `Type { key { fields } key: ConcreteType { fields } }`

顶层 `id` 保留名不允许作为字段名。分组类型可以是抽象父类，但 `key {}` 只能用于非抽象具体类型；抽象分组必须用 `key: ConcreteType {}`。

- [ ] **Step 2: 分隔符迁移**

字段、数组、字典统一使用逗号分隔，允许尾逗号。记录块结束由 `}` 决定，不需要分号。去掉 `=` 字段赋值兼容；字段值 separator 必须是 `:`。

- [ ] **Step 3: 引用解析**

`parse_value` 在对象上下文接受：

- `&key`：用当前期望对象类型生成 `record_ref(expected_type, key)`。
- `@Type.key`：生成 `record_ref(Type, key)`。
- `@Type.key.path[index]`：生成 `path_ref(Type, key, segments)`。

`@key` 报错，`&key.path` 报错，裸 `key` 在对象字段里报 `ReferenceNeedsMarker`。

- [ ] **Step 4: `...` 解析**

对象内字段位置支持 `...value`，产出 `CfdInputValue::object_spread`。字典条目位置支持 `...value`，产出 `CfdInputValue::dict_spread`。spread 值使用当前对象或字典类型上下文解析，因此允许 `&key`、`@Type.key.path`、inline object/dict。

- [ ] **Step 5: 字符串和注释**

保留当前双引号字符串和简单裸字符串能力。注释支持 `#`；可以继续接受 `//` 作为兼容输入，但 spec 只记录 `#`。

Expected: all CFD tests pass.
