# 词法分析器实现计划

本文整理 coflow 核心版本需要进入词法分析器的 token 集合和实现边界。实现语言使用 Rust，lexer crate 使用最新稳定版 `logos = "0.16.1"`。

核心依据：

1. `docs/0-core-language.md`
2. `docs/1-core-types.md`
3. `docs/2-core-config.md`
4. `docs/3-core-runtime.md`

## 目标

第一阶段只实现核心语言的词法分析，不把 `docs/proposals/` 中的提案语法纳入正式 token 语义。

lexer 输出应包含：

1. `TokenKind`：token 类型。
2. `Span`：源码字节区间，用于诊断和 parser。
3. 原始切片或可延迟解析的字面量信息。
4. 词法错误，包含错误位置和错误种类。

lexer 不负责：

1. 解析表达式优先级。
2. 判断顶层语法是否合法。
3. 判断 `yield break`，`yield from`，`co fn` 的上下文合法性。
4. 判断 `{ ... }` 是对象还是字典。
5. 执行字符串转义或数值范围检查之外的语义判断。

## 依赖

`Cargo.toml`：

```toml
[dependencies]
logos = "0.16.1"
unicode-ident = "1"
```

建议先暴露稳定的项目内 API，不让 parser 直接依赖 `logos::Lexer`：

```rust
pub struct Token {
    pub kind: TokenKind,
    pub span: Span,
}

pub struct Span {
    pub start: usize,
    pub end: usize,
}
```

## Token 分类

### 关键字

核心版本需要以下关键字：

| 源码 | TokenKind | 说明 |
| --- | --- | --- |
| `import` | `Import` | 模块导入 |
| `as` | `As` | import 别名 |
| `local` | `Local` | 文件内私有声明 |
| `class` | `Class` | class 声明 |
| `enum` | `Enum` | enum 声明 |
| `validate` | `Validate` | class 内校验块 |
| `fn` | `Fn` | 普通函数或匿名函数 |
| `co` | `Co` | `co fn` 的前缀 |
| `var` | `Var` | 变量声明 |
| `if` | `If` | 条件语句 |
| `else` | `Else` | 条件分支 |
| `while` | `While` | while 循环 |
| `for` | `For` | for 循环 |
| `in` | `In` | for-in 和成员判断 |
| `break` | `Break` | 跳出循环 |
| `continue` | `Continue` | 继续循环 |
| `return` | `Return` | 函数返回 |
| `throw` | `Throw` | 抛出运行时错误 |
| `try` | `Try` | try 块 |
| `catch` | `Catch` | catch 块 |
| `yield` | `Yield` | co fn 产出 |
| `from` | `From` | `yield from` |
| `and` | `And` | 逻辑与 |
| `or` | `Or` | 逻辑或 |
| `not` | `Not` | 逻辑非 |
| `true` | `True` | bool 字面量 |
| `false` | `False` | bool 字面量 |
| `null` | `Null` | 空字面量和 null 类型名 |
| `self` | `SelfKw` | validate 中的当前对象 |

说明：

1. `from` 在核心版本中不是 import 语法，只用于 `yield from`。lexer 仍把它作为关键字，parser 负责限制上下文。
2. `co fn` 不需要合并成一个 token。保留 `Co` + `Fn`，parser 检查二者相邻关系。
3. `yield break` 和 `yield from` 不需要合并成 token。保留 `Yield` + `Break` / `From`。
4. 基础类型名 `int`，`float`，`bool`，`string`，`any` 可以先作为普通 `Ident`，由类型解析阶段识别。`null` 同时是字面量和类型名，建议保留为 `Null`。

### 标识符

| TokenKind | 规则 |
| --- | --- |
| `Ident` | Unicode XID 标识符，允许 `_` 开头 |

coflow 对 Unicode 友好，标识符应支持中文和其他语言文字。第一阶段建议采用 Unicode Standard Annex #31 的 XID 规则：

1. 首字符允许 `_` 或 `XID_Start`。
2. 后续字符允许 `_` 或 `XID_Continue`。
3. ASCII 标识符仍然是该规则的子集。
4. emoji，空格，标点和运算符不允许出现在标识符中。

示例：

```coflow
var 玩家生命 = 100
var скорость = 3.5
var Δx = 1
fn 计算伤害(基础伤害, 倍率) {
  return 基础伤害 * 倍率
}
```

Rust 实现上有两种可选方式：

1. 优先尝试在 `logos` regex 中使用 Unicode 属性：`[_\p{XID_Start}][_\p{XID_Continue}]*`。
2. 如果 `logos` regex 对目标属性支持不够稳定，则用较宽的 Unicode regex 捕获候选标识符，再用 `unicode-ident` 的 `is_xid_start` / `is_xid_continue` 做二次校验。

关键字优先于 `Ident` 匹配。

关键字本身仍只使用 ASCII 拼写，例如 `fn`，`class`，`return`。非 ASCII 名字不会和关键字冲突。

## 字面量

### 整数

| TokenKind | 示例 | 规则 |
| --- | --- | --- |
| `IntLiteral` | `0`, `10`, `1_000`, `0xff`, `0b1010`, `0o755` | 整数字面量 |

核心版本支持数字分隔符和前缀进制整数字面量：

| 形式 | 示例 | 说明 |
| --- | --- | --- |
| 十进制 | `123`, `1_000` | 默认整数形式 |
| 十六进制 | `0xff`, `0xFF`, `0xdead_beef` | 前缀 `0x` 或 `0X` |
| 二进制 | `0b1010`, `0b1111_0000` | 前缀 `0b` 或 `0B` |
| 八进制 | `0o755`, `0o12_34` | 前缀 `0o` 或 `0O` |

数字分隔符规则：

1. `_` 只能出现在两个合法数字之间。
2. `_` 不能出现在开头或结尾。
3. `_` 不能紧跟进制前缀，例如 `0x_ff` 非法。
4. `_` 不能连续出现，例如 `1__000` 非法。

整数字面量规则：

1. 不把负号纳入数值字面量，`-1` 由 `Minus` + `IntLiteral` 组成。
2. lexer 可以先识别带前缀的整数 token，再由字面量解析阶段去掉 `_` 并转换进制。
3. 非法数字字符应产生词法错误或字面量错误，例如 `0b102`，`0xzz`。

### 浮点数

| TokenKind | 示例 | 规则 |
| --- | --- | --- |
| `FloatLiteral` | `3.5`, `0.5`, `10.0` | 至少包含小数点且小数点后至少一位数字 |

第一阶段建议：

1. 支持 `1.0`，`0.5`，`1_000.5`，`1.000_5`。
2. 不支持 `.5`，避免和字段访问 `.` 冲突。
3. 不支持 `1.`，避免错误吞掉后续 token。
4. 不支持科学计数法，后续需要时再加。

浮点数字分隔符同样只能出现在两个数字之间。`1_.0`，`1._0`，`1.0_` 非法。

匹配顺序上，`FloatLiteral` 应优先于 `IntLiteral`。

### 字符串

核心版本支持四类字符串：

| TokenKind | 示例 | 说明 |
| --- | --- | --- |
| `StringLiteral` | `"hero"` | 普通双引号字符串，处理转义 |
| `RawStringLiteral` | `r"C:\game\hero.png"` | 原始单行字符串，不处理转义 |
| `MultilineStringLiteral` | `"""line"""` | 三引号多行字符串，保留换行 |
| `RawMultilineStringLiteral` | `r"""line"""` | 原始三引号多行字符串 |

普通字符串第一阶段支持的转义：

| 源码 | 含义 |
| --- | --- |
| `\"` | 双引号 |
| `\\` | 反斜杠 |
| `\n` | 换行 |
| `\r` | 回车 |
| `\t` | 制表 |

第一阶段不支持字符串插值。`${...}` 在字符串内部只是普通字符，或由字符串解码阶段按规则拒绝。

实现建议：

1. `logos` 负责识别完整字符串 token 范围。
2. 单独写字符串解码函数处理转义和错误。
3. 未闭合字符串返回词法错误。
4. 块注释和字符串内的注释符号不参与注释识别。

## 运算符

### 赋值运算符

| 源码 | TokenKind | 说明 |
| --- | --- | --- |
| `=` | `Eq` | 普通赋值或配置定义 |
| `+=` | `PlusEq` | 复合赋值 |
| `-=` | `MinusEq` | 复合赋值 |
| `*=` | `StarEq` | 复合赋值 |
| `/=` | `SlashEq` | 复合赋值 |
| `%=` | `PercentEq` | 复合赋值 |
| `??=` | `QuestionQuestionEq` | 空值合并赋值 |

### 算术运算符

| 源码 | TokenKind |
| --- | --- |
| `+` | `Plus` |
| `-` | `Minus` |
| `*` | `Star` |
| `/` | `Slash` |
| `%` | `Percent` |

### 比较运算符

| 源码 | TokenKind |
| --- | --- |
| `==` | `EqEq` |
| `!=` | `BangEq` |
| `<` | `Lt` |
| `<=` | `LtEq` |
| `>` | `Gt` |
| `>=` | `GtEq` |

说明：核心语言使用 `not` 做逻辑非，但 `!=` 仍是比较运算符。单独的 `!` 不进入核心版本 token。

### 空值合并

| 源码 | TokenKind |
| --- | --- |
| `??` | `QuestionQuestion` |

`??=` 作为正式 token 支持。parser 后续负责限定它只能出现在可赋值目标上。

### 成员与访问

| 源码 | TokenKind | 说明 |
| --- | --- | --- |
| `.` | `Dot` | 字段访问和 enum 值访问 |
| `?.` | `QuestionDot` | 可选字段访问 |

`?.` 作为正式 token 支持。parser 后续负责限制可选访问的语义和优先级。核心版本仍不支持 `..`。

## 分隔符

| 源码 | TokenKind | 说明 |
| --- | --- | --- |
| `(` | `LParen` | 调用和分组 |
| `)` | `RParen` | 调用和分组 |
| `{` | `LBrace` | 块，对象，字典 |
| `}` | `RBrace` | 块，对象，字典 |
| `[` | `LBracket` | 数组，索引，数组类型，字典类型 |
| `]` | `RBracket` | 数组，索引，数组类型，字典类型 |
| `,` | `Comma` | 参数，元素，字段分隔 |
| `:` | `Colon` | 类型标注，对象/字典字段，字典类型 |

核心文档示例允许对象和数组尾随逗号。lexer 只产出 `Comma`，是否允许尾随由 parser 决定。

## 注释和空白

### 空白

跳过以下空白：

1. 空格
2. tab
3. `\r`
4. `\n`
5. form feed

核心语言不使用换行作为语句分隔 token，因此 lexer 可直接跳过换行。

### 注释

| 形式 | 说明 |
| --- | --- |
| `// ...` | 单行注释，到行尾结束 |
| `/* ... */` | 块注释，不支持嵌套 |

实现建议：

1. 注释默认跳过，不输出 token。
2. 未闭合块注释返回词法错误。
3. 嵌套块注释不特殊处理，遇到第一个 `*/` 结束。

## 非核心语法暂不进入

以下语法已在核心文档中明确排除，第一阶段不作为正式 token 语义支持：

| 语法 | 处理方式 |
| --- | --- |
| `from import` | `from` 只作为 `yield from` 关键字 |
| `match` | 作为 `Ident` |
| `is` | 作为 `Ident` |
| `not in` | 保持 `Not` + `In`，parser 不接受为单个成员判断 |
| `..` / range | 两个 `Dot`，parser 报错 |
| 切片 | 不提供额外 token |
| 解构 | 不提供额外 token |
| 展开 `...` | 三个 `Dot`，parser 报错 |
| 字符串插值 | 字符串解码阶段拒绝或保持普通文本 |
| 命名参数和默认参数 | token 足够，parser 不接受对应语法 |
| 动态 key 字面量 | token 足够，parser 不接受对应语法 |
| 联合类型 `A | B` | 不提供 `Pipe` token，遇到 `|` 报词法错误 |
| 可空类型糖 `T?` | 不提供 `Question` token，遇到单个 `?` 报词法错误 |

## 词法错误

建议第一阶段定义以下错误：

```rust
pub enum LexErrorKind {
    UnexpectedChar,
    UnterminatedString,
    UnterminatedBlockComment,
    InvalidEscape,
    InvalidNumber,
}
```

说明：

1. `InvalidEscape` 可以在字符串解码阶段产生，不一定由 `logos` 直接产生。
2. 数值溢出可先放到字面量解析阶段处理。
3. 非核心符号如 `|`，`?`，`!` 可归为 `UnexpectedChar`，但 `!=` 要正常识别。
4. `InvalidNumber` 用于非法进制数字和非法 `_` 位置。

## 实现顺序

1. 建立 Rust crate 基础结构。
2. 引入 `logos = "0.16.1"`。
3. 实现 `TokenKind` 与 `Token`。
4. 实现 keyword，identifier，number，operator，delimiter 的识别。
5. 实现单行注释和普通空白跳过。
6. 实现块注释跳过和未闭合块注释错误。
7. 实现四类字符串 token 识别。
8. 实现字符串解码和转义错误。
9. 实现数值字面量二次校验，包括前缀进制和数字分隔符。
10. 补充 span 单元测试。
11. 用核心文档中的代码片段建立 lexer fixture 测试。

## 测试清单

测试应分为正例 token 序列、span、错误三类。错误测试需要断言错误种类和错误 span，不能只断言“失败”。

### 关键字和标识符

1. 所有关键字能被识别为对应 token。
2. 关键字前后有标识符字符时必须是 `Ident`，例如 `imported`，`className`，`return_value`。
3. 类型名 `int`，`float`，`bool`，`string`，`any` 识别为 `Ident`。
4. `null` 识别为 `Null`，不识别为 `Ident`。
5. `co fn` 产出 `Co` + `Fn`。
6. `yield from` 产出 `Yield` + `From`。
7. `yield break` 产出 `Yield` + `Break`。
8. `for item in items` 产出 `For` + `Ident` + `In` + `Ident`。
9. `item in items` 中的 `in` 仍为 `In`。

### Unicode 标识符

1. 中文标识符能被识别，例如 `玩家生命`，`计算伤害`。
2. 非中文 Unicode 标识符能被识别，例如 `скорость`，`Δx`，`名前`。
3. Unicode 标识符允许后续数字，例如 `速度2`，`变量_1`。
4. `_` 可以作为开头，例如 `_临时值`，`_1`。
5. 数字开头不能整体识别为标识符，例如 `1变量` 应报 `InvalidNumber`，避免静默切成 `IntLiteral` + `Ident`。
6. emoji 不能进入标识符，例如 `hp😀` 应产出 `Ident` 后在 emoji 位置报 `UnexpectedChar`。
7. 零宽字符、空格和标点不能进入标识符。

### 运算符 longest-match

1. `??=` 必须识别为 `QuestionQuestionEq`，不能拆成 `QuestionQuestion` + `Eq`。
2. `??` 必须识别为 `QuestionQuestion`。
3. `?.` 必须识别为 `QuestionDot`，不能拆成错误 `?` + `Dot`。
4. `+=`，`-=`，`*=`，`/=`，`%=` 必须识别为对应复合赋值 token。
5. `==`，`!=`，`<=`，`>=` 必须优先于单字符 token。
6. 单字符 `=`，`+`，`-`，`*`，`/`，`%`，`<`，`>`，`.` 正常识别。
7. `/=` 必须优先于 `/`，`//` 必须作为单行注释，`/*` 必须作为块注释开始。
8. `...` 在核心版本中识别为三个 `Dot`，由 parser 报非核心语法错误。
9. `..` 识别为两个 `Dot`，由 parser 报非核心范围语法错误。
10. 单独 `?` 报 `UnexpectedChar`。
11. 单独 `!` 报 `UnexpectedChar`，但 `!=` 必须合法。
12. `|`，`&`，`^` 报 `UnexpectedChar`。

### 分隔符和结构

1. `(`，`)`，`{`，`}`，`[`，`]`，`,`，`:` 均能识别。
2. 对象字面量 `{ id: "sword", damage: 10, }` token 序列正确。
3. 字典类型 `[string: int]` token 序列正确。
4. 数组类型 `[int]` 和数组字面量 `[1, 2, 3]` token 序列正确。
5. 空对象 `{}`，空数组 `[]`，空参数 `()` token 序列正确。

### 整数字面量

1. 十进制整数：`0`，`10`，`123`，`1_000_000`。
2. 十六进制整数：`0xff`，`0XFF`，`0xdead_beef`。
3. 二进制整数：`0b0`，`0b1`，`0b1010_0101`。
4. 八进制整数：`0o0`，`0o755`，`0O12_34`。
5. 负数不属于单个数字 token，`-1` 必须产出 `Minus` + `IntLiteral`。
6. 非法分隔符报 `InvalidNumber`：`1_`，`1__0`，`0_x`，`0x_ff`，`0b_1010`，`0o755_`。
7. 非法进制数字报 `InvalidNumber`：`0b102`，`0o789`，`0xzz`。
8. 只有进制前缀没有数字时报 `InvalidNumber`：`0x`，`0b`，`0o`。
9. 前缀大小写都要覆盖：`0X1F`，`0B1010`，`0O755`。
10. 数字后直接跟标识符起始字符报 `InvalidNumber`：`123abc`，`1变量`。

### 浮点字面量

1. 合法浮点：`1.0`，`0.5`，`10.25`，`1_000.5`，`1.000_5`。
2. `1.` 不应识别为合法浮点；建议产出 `IntLiteral` + `Dot`，由 parser 处理后续语法。
3. `.5` 不应识别为合法浮点；应产出 `Dot` + `IntLiteral`。
4. `-1.5` 必须产出 `Minus` + `FloatLiteral`。
5. 非法分隔符报 `InvalidNumber`：`1_.0`，`1._0`，`1.0_`，`1__0.0`，`1.0__0`。
6. 科学计数法暂不支持：`1e3`，`1.0e-3`，`1E3` 应报 `InvalidNumber`，不能静默识别成多个合法 token。

### 字符串字面量

1. 普通字符串：`"hero"`，`""`。
2. 普通字符串合法转义：`\"`，`\\`，`\n`，`\r`，`\t`。
3. 普通字符串非法转义报 `InvalidEscape`：`\x`，`\u`，`\0`。
4. 普通字符串遇到未转义换行时报 `UnterminatedString`。
5. 普通字符串未闭合时报 `UnterminatedString`。
6. 原始字符串：`r"C:\game\assets\hero.png"`，反斜杠不触发转义。
7. 原始字符串中 `\"` 不作为转义；如果遇到 `"`，字符串结束。
8. 多行字符串 `"""line one\nline two"""` 保留换行。
9. 原始多行字符串 `r"""...\n..."""` 保留内容且不处理转义。
10. 空多行字符串 `""""""` 能识别。
11. 未闭合多行字符串和原始多行字符串报 `UnterminatedString`。
12. 字符串内的 `//` 和 `/* */` 不触发注释。
13. 字符串内的 `${name}` 不进入插值模式。
14. `r"..."` 必须识别为 `RawStringLiteral`，不能识别为 `Ident("r")` + `StringLiteral`。
15. `r"""..."""` 必须识别为 `RawMultilineStringLiteral`，且优先于 `RawStringLiteral`。
16. `r """..."""` 中间有空白时，必须识别为 `Ident("r")` + `MultilineStringLiteral`。

### 注释和空白

1. 空格，tab，`\r`，`\n`，form feed 被跳过。
2. 单行注释 `// comment` 跳过到行尾。
3. 文件末尾的单行注释没有换行也合法。
4. 块注释 `/* comment */` 被跳过。
5. 块注释可跨多行。
6. 未闭合块注释报 `UnterminatedBlockComment`。
7. 块注释不嵌套：`/* a /* b */ c */` 在第一个 `*/` 结束，后续 `c */` 按普通 token/错误处理。
8. 注释内的关键字、字符串和符号不产出 token。
9. `/ /` 不是注释，应产出两个 `Slash`。
10. `/= // comment` 应产出 `SlashEq`，然后跳过单行注释。

### Span 和错误恢复

1. 每个 token 的 `span` 必须覆盖原始源码字节区间。
2. Unicode 标识符 span 使用字节索引，不能按字符数错位。
3. 被跳过的空白和注释不产出 token，但后续 token span 必须正确。
4. 错误 span 指向导致错误的最小源码区间，例如单独 `?` 的 span 只覆盖 `?`。
5. 一个输入中多个错误应尽量全部报告，例如 `? | !` 应报告三个 `UnexpectedChar`。
6. EOF 不由 `logos` 匹配；如 parser 需要，lexer 包装层追加 `Eof`，span 为源码末尾空区间。

## 待确认点

以下点会影响 token 规则，建议在 parser 开始前确认：

1. 是否需要科学计数法浮点数，例如 `1e-3`。
2. 普通字符串是否支持 Unicode 转义，例如 `\u{1F600}`。
3. 未闭合字符串是否由 `logos` 直接报错，还是由手写扫描函数包一层处理。
